# OAuth2 Setup for docuware-client

DocuWare 7.10+ uses OAuth2 for authentication. The library supports four
auth flows; three of them need an entry in **DocuWare Configuration →
Integrations → App Registration** before they can be used.

## Picking an auth flow

| Flow | When to use | App Registration type |
|---|---|---|
| Password Grant | Legacy / existing setups, internal scripts using a real DocuWare user account | none (uses the built-in OAuth2 password grant against an existing user) |
| Native (PKCE) | Desktop apps, CLI tools that can open a browser on the same machine | **Native application** |
| Web Application (PKCE + secret) | Server-side apps where users log in via the browser; you want a stronger client identity check than PKCE alone | **Web application** |
| Service / Client Credentials | Backend jobs, ETL, MCP servers, scheduled tasks — no human user involved | **Trusted / Service application** |

Password Grant is increasingly considered legacy by the OAuth2 spec (RFC
6749 §4.3 carries a security warning, OAuth 2.1 removes it entirely). New
integrations should prefer one of the other three.

## Native application (PKCE)

For desktop apps and CLI tools that can open the system browser.

### Register the app in DocuWare

DocuWare Configuration → Integrations → App Registration → **Add**:

| Field | Value |
|---|---|
| App type | **Native application** |
| Grant type | Authorization Code with PKCE |
| Client ID | UUID assigned by DocuWare on save — *copy it* |
| URL Redirects | `http://localhost:18923/callback` (port must match your code exactly) |
| Scopes | `docuware.platform  openid  dwprofile  offline_access` |
| Refresh token | enabled |

Click **Save** — the registration is inactive until saved. An unsaved
registration produces `invalid_request: Invalid redirect_uri` errors.

### Use it from Python

```python
import pathlib
import docuware

client = docuware.connect(
    url="acme.docuware.cloud",
    authenticator=docuware.PkceAuthenticator(
        client_id="<UUID from App Registration>",
        redirect_port=18923,
    ),
    credential_store=docuware.JsonFileCredentialStore(
        pathlib.Path.home() / ".config" / "docuware-client" / ".credentials"
    ),
)
```

First call opens the browser. Subsequent calls reuse the persisted tokens.
Refresh tokens rotate automatically and are persisted on every refresh
(RFC 6749 §10.4 token-family invalidation is handled).

See `examples/oauth2_login.py` for a complete runnable script.

## Web application (PKCE + client_secret)

For server-side apps where the redirect URI is a real public URL or the
app needs a stronger client identity than PKCE alone provides.

Same as Native, but the App Registration produces an additional
**Client Secret**, and the redirect URI typically points to your web
server (not localhost). Pass the secret to `PkceAuthenticator`:

```python
authenticator=docuware.PkceAuthenticator(
    client_id="...",
    client_secret="...",
    redirect_port=443,   # or whatever your web server listens on
)
```

See `examples/oauth2_webapp.py`.

## Service / Trusted application (Client Credentials)

For backends that talk to DocuWare under their own service identity, no
human user involved. No browser, no callback.

### Register the app in DocuWare

DocuWare Configuration → Integrations → App Registration → **Add**:

| Field | Value |
|---|---|
| App type | **Trusted application** (or "Service application", DocuWare naming varies) |
| Grant type | Client Credentials |
| Client ID | UUID assigned on save |
| Client Secret | generated by DocuWare on save — *copy it before closing the dialog* |
| Scopes | `docuware.platform` is enough |

### Use it from Python

```python
import docuware

client = docuware.connect(
    url="acme.docuware.cloud",
    authenticator=docuware.ClientCredentialsAuthenticator(
        client_id="...",
        client_secret="...",
    ),
    credential_store=docuware.JsonFileCredentialStore("/etc/myapp/.credentials"),
)
```

Client Credentials does **not** issue refresh tokens (RFC 6749 §4.4.3). On
401/403 the library re-acquires a fresh access token from the stored
secret pair — that re-acquisition is the substitute for refresh.

See `examples/oauth2_client_credentials.py`.

## The credential store

`JsonFileCredentialStore` is the library default — a JSON file with mode
`0o600` and atomic writes. The file lives wherever you point it, typically
`~/.config/docuware-client/.credentials`. The shape:

```json
{
    "method": "pkce",
    "url": "https://acme.docuware.cloud",
    "client_id": "...",
    "access_token": "...",
    "refresh_token": "...",
    "token_endpoint": "..."
}
```

The `method` field discriminates between the four flows. If absent on
load, the library assumes `"password"` — that keeps existing `.credentials`
files from `docuware-client < 0.8.0` working unchanged.

If you need to store credentials somewhere else (OS keyring, secrets
manager, encrypted DB row), subclass `CredentialStore` and implement
`load()` / `save()` — atomic. The library calls `save()` once after the
initial login and after every successful token refresh.

## Troubleshooting

**`invalid_request: Invalid redirect_uri`**
The redirect URI in your code does not byte-match what's registered in
DocuWare. Port matters; trailing slash matters. Save the App Registration
after editing — unsaved changes don't take effect.

**`invalid_grant`** during the first PKCE callback
The authorization code expired (default ~60 s) or was already used. Just
re-run; the flow generates a fresh code.

**`invalid_grant`** on a refresh (PKCE)
The refresh token was already invalidated (token-family revocation per
RFC 6749 §10.4). The library normally persists rotated tokens to your
`CredentialStore` automatically — if you see this, check that `save()`
in your store is actually atomic and reachable.

**`401` repeatedly on Client Credentials**
The `client_secret` is wrong, or the App Registration was not saved /
not activated. There is no refresh path here — every 401 triggers a full
re-acquisition, so a bad secret produces a hot loop. Fix the secret.

**Token file accidentally world-readable**
`JsonFileCredentialStore` sets `0o600` on every save. If you see `0o644`,
check that nothing else (your editor, a sync tool) is rewriting the file.

## DocuWare-specific notes

DocuWare's IdentityServer also advertises grant types like `dwtoken`,
`trusted`, `impersonation`, `delegation`, `windows`, `device_code`, and
`ciba`. The library currently supports only the four covered above.
Device Code Grant (for CLI/headless without a browser) is on the roadmap.
